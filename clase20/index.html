¿ChatGPT puede crear la API de los alumnos?

Puede ayudar a generar y explicar server.js, el endpoint, el prompt y las pruebas. Pero el alumno debe guardar los archivos, instalar dependencias y ejecutar el servidor. ChatGPT no convierte una conversación en una API pública ni aporta automáticamente crédito de OpenAI. En esta clase, la API propia es Express y el proveedor principal es Ollama.

Misión 1

Comprendan la arquitectura que construirán

El frontend recibe y presenta datos. Express valida, protege la configuración, arma el prompt y conserva el contrato. Ollama procesa la solicitud localmente. El navegador nunca se conecta directamente al proveedor.

  1. El usuario escribe su situación.
  2. JavaScript valida el formulario.
  3. fetch() envía POST a /api/agent.
  4. Express recibe y valida el body.
  5. El backend agrega el prompt del sistema.
  6. El servidor consulta Ollama.
  7. El modelo devuelve texto.
  8. El backend extrae y normaliza JSON.
  9. El frontend recibe JSON.
  10. La página presenta el resultado.

Antes de avanzar, explíquenlo con sus propias palabras

¿Qué dato escribe el usuario, qué ruta recibe la solicitud y qué resultado debe presentar la página? Si el equipo puede recorrer esos tres puntos sobre el diagrama, está listo para crear los archivos.

Conceptos del servidor

Misión 5

Diseñen el contrato de la API

La regla compartida del sistema

¿Qué es un contrato?

Es el acuerdo que define qué debe enviar el frontend y qué promete devolver el backend. No es un documento legal ni una función adicional: es la forma estable de los datos que permite que dos partes programadas por separado se entiendan.

Una analogía

Piensen en un contacto eléctrico: el enchufe y la toma funcionan juntos porque coinciden en forma y conexión. Si el frontend espera answer pero Express devuelve respuesta, ambos archivos pueden estar “bien” por separado y aun así el producto no mostrará nada.

1

El frontend lo usa

Para saber qué propiedades incluir en JSON.stringify() y qué campos leer al presentar el resultado.

2

Express lo protege

Para validar tipos, rechazar entradas incorrectas y normalizar la salida del modelo.

3

Ollama se adapta

Puede producir texto variable, pero el backend lo transforma a la estructura estable que la interfaz conoce.

¿Por qué definirlo antes de seguir programando?

  • Evita que frontend y backend inventen nombres distintos para el mismo dato.
  • Permite construir y probar la interfaz aunque el modelo todavía no responda.
  • Hace posible cambiar Ollama por otro proveedor sin rediseñar la página.
  • Define qué errores debe poder interpretar el usuario.
  • Reduce cambios en cadena: prompt, servidor, interfaz e historial trabajan sobre la misma estructura.

Request · lo que entra

message contiene la consulta. context agrega datos opcionales que ayudan a interpretar la solicitud.

Response 200 · lo que sale

ok confirma el resultado; result contiene datos para la interfaz y meta describe la operación.

Anatomía del contrato base

message
Texto principal que el usuario desea procesar.
context
Objeto opcional con información complementaria; nunca debe considerarse confiable sin validarlo.
ok
Indica si la API completó la operación según el contrato.
result
Objeto estable que la interfaz puede presentar sin interpretar texto libre.
meta
Información técnica no sensible, como la fecha y el proveedor.
error
Mensaje claro cuando la operación no puede completarse.

Exploren el request sin llamar al modelo

Cambien el mensaje y el nombre del proyecto. La herramienta solo muestra cómo quedaría el JSON; todavía no envía información a Express ni a Ollama.

El contrato también incluye los errores

El status HTTP explica la categoría del resultado; el JSON explica qué ocurrió con palabras comprensibles. El frontend usa ambos para decidir qué mostrar.

200 respuesta correcta400 entrada inválida404 ruta inexistente429 límite temporal500 error interno502 Ollama no respondió correctamente
{
  "ok": false,
  "error": "Escribe una consulta antes de continuar."
}

Qué ocurre si no coinciden

Backend devuelve: { "respuesta": "..." }

Frontend intenta leer: data.result.answer

Resultado: la respuesta existe en la red, pero la pantalla queda vacía o produce un error.

Cuando cambien un campo del contrato, busquen ese nombre en SYSTEM_PROMPT, normalizeResult(), renderResult() y el historial. Los cuatro lugares deben seguir hablando el mismo idioma.

Definan la personalidad mínima antes de copiar el servidor completo

  • Nombre: cómo se identifica el producto.
  • Usuario: quién lo utilizará.
  • Propósito: una tarea concreta.
  • Reglas: qué criterios debe aplicar.
  • Límites: qué no puede decidir o afirmar.
  • Salida: campos JSON que la interfaz sabrá presentar.

En la Misión 12 profundizarán la personalización visual, pero estos elementos ya son necesarios para completar SYSTEM_PROMPT en la siguiente misión.

La misión está lista cuando el equipo puede responder:
  1. ¿Qué envía el navegador?
  2. ¿Qué valida Express?
  3. ¿Qué estructura recibe de regreso la interfaz?
  4. ¿Qué respuesta llega cuando algo falla?
  5. ¿Qué archivos deben cambiar juntos si renombramos un campo?
Misión 6

Conecten el servidor con la inteligencia artificial

El endpoint valida la entrada, arma el prompt, consulta /api/chat de Ollama y transforma la salida en el contrato estable del producto.

Secuencia exacta
  1. Detengan Express con Ctrl + C.
  2. Reemplacen todo el contenido de server.js por la versión siguiente.
  3. Sustituyan los textos entre corchetes del SYSTEM_PROMPT.
  4. Guarden el archivo.
  5. Ejecuten npm run check y después npm run dev.
Prueba curl
curl -X POST http://localhost:3000/api/agent \
  -H "Content-Type: application/json" \
  -d '{"message":"Necesito orientación sobre mi situación"}'
PowerShell
Invoke-RestMethod -Uri http://localhost:3000/api/agent `
  -Method Post `
  -ContentType "application/json" `
  -Body '{"message":"Necesito orientación sobre mi situación"}'
Qué deben entender

fetch del servidor consulta Ollama; system contiene las reglas; format: "json" solicita JSON; el fallback conserva una respuesta útil; los códigos HTTP distinguen errores del alumno, servidor y proveedor. El usuario nunca recibe el error técnico completo.

Misión 7

Construyan la interfaz del producto

La interfaz base es completa pero debe adaptarse a la tarea. Un recomendador puede usar tarjetas; un clasificador, categorías; un orientador, pasos.

Abran la aplicación desde Express

Usen http://localhost:3000 ↗. No abran public/index.html con doble clic y no utilicen Live Server: fetch('/api/agent') necesita que Express sirva frontend y API desde el mismo origen.

Personalicen esta interfaz

  • Nombre, logotipo y descripción.
  • Texto, ejemplos y campos del formulario.
  • Forma de presentar resultados y terminología.
  • Colores y jerarquía del dominio.

No todos los proyectos deben parecer chat.
